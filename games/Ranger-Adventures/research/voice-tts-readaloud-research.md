# Ranger van de Veluwe — Voice / Text-to-Speech for the Read-Aloud (Karaoke) Feature (Research Report, 22 June 2026)

Scope: a **private, non-commercial** iPad-first (Safari) 3D web game for an 8-year-old Dutch boy with dyslexia. Every instruction, story beat and fact is **spoken in Dutch (nl-NL)** with a **word-by-word "karaoke" highlight**. Hard constraints from the repo: **no third-party scripts/trackers, all data stays client-side, free/open tools only** (CC0/CC BY/NC/MIT/Apache/GPL all acceptable because it is private + non-commercial). Confidence tags: **zeker** = verified against a primary source; **waarschijnlijk** = likely from secondary/practitioner evidence; **onzeker** = unverified, must test.

## TL;DR
- **Ship now:** keep the **browser Web Speech API (`speechSynthesis`)** but **steer it to a downloaded Apple "Enhanced/Premium" Dutch voice** — on iPad that is **Xander (nl-NL, standaard ABN)** via *Instellingen → Toegankelijkheid → Gesproken materiaal → Stemmen → Nederlands → Xander → Verbeterde/Premium kwaliteit*. This is free, fully on-device, offline, and removes the "weak default voice" complaint. **The voice is a swappable layer — design the read-aloud around an interface, not around `speechSynthesis` directly.** *zeker* the voice exists; *waarschijnlijk* it materially beats the default.
- **Karaoke word-sync on iPad Safari is the hard part and the answer is: do NOT trust `onboundary` for word timing on iOS.** macOS Safari fires word-level boundary events, but iOS/iPadOS support is inconsistent/unreliable and the event is missing fields (`charLength`) even where it fires. **Build your own word-timing instead** — a per-word duration estimator (length/syllable-weighted), which the current prototype already approximates with its "timed fallback." That fallback should be the *primary* karaoke driver on iPad, not the backup. *waarschijnlijk* (practitioner consensus; needs a 5-minute on-device confirm).
- **Best free/open upgrade (better Dutch than Apple, still €0):** **Piper** neural voices (MIT) — but **use the Belgian-Dutch (nl_BE) voices (nathalie / rdh), not nl_NL-mls**, which is widely reported as "gibberish" on short sentences. In-browser Piper (ONNX/WASM via `@mintplex-labs/piper-tts-web`) is *possible* but **risky on iPad Safari** because ONNX-Runtime-Web has open, unresolved iOS bugs (WASM-SIMD wrong output on iOS 16.4+; severe CPU/RAM blow-up in WebGPU/JSEP mode on Safari 26.x). The robust pattern is **build-time pre-bake**: render Piper audio + word-timing to static files at build, ship `.mp3/.opus` + a timing JSON. *zeker* on license/voices/bugs; *waarschijnlijk* on in-browser feasibility.
- **Voice-cloning the parent's voice — verdict: feasible and license-clean for THIS private project, but only as a build-time pre-bake, never realtime in the browser, and it cannot cover the dynamic name slots.** Best pick: **XTTS-v2** (supports Dutch, ~6 s of reference audio, best open quality) under the **Coqui Public Model License (CPML) — strictly NON-COMMERCIAL, but no EU/region exclusion**, so it is fine here. The fully-permissive alternative is **OpenVoice V2 (MIT)**, but its Dutch is only cross-lingual (not a native nl voice) so quality is less certain. **Apple "Personal Voice" is the cleanest cloning tech but is English+Mandarin only and is a native-iOS API not reachable from a Safari web game — so it does not solve this.** *zeker* on licenses; *waarschijnlijk* on Dutch realism.
- **Kokoro is out for Dutch.** The Kokoro-82M model supports 9 languages and **Dutch is not one of them** (and the browser `kokoro.js` build is English-only). Do not plan on it for nl-NL. *zeker.*
- **License red flags to remember:** **F5-TTS** model weights are **CC-BY-NC** (code is MIT) — NC-only; **XTTS-v2 = CPML, NC-only**; **Tortoise/most "in-the-wild"-trained models = NC**. All are *fine for a private non-commercial kids' app* but would block any future commercial release. eSpeak-NG / MaryTTS are permissive but MaryTTS **has no Dutch** and eSpeak-NG Dutch is robotic (quality floor only). *zeker.*

## Key Findings

### A. The architecture decision dominates everything: treat voice as a swappable layer with two independent jobs
The read-aloud has **two separable problems**: (1) *produce audio* for a Dutch string, and (2) *know when each word is spoken* for the highlight. Different engines solve these differently, and iPad Safari constrains both. The right design is a small `Narrator` interface with two methods — `speak(text)` and a `wordTimings(text)` source — so you can swap the audio engine (browser voice → pre-baked Piper → cloned parent voice) without touching the karaoke renderer. The dynamic-name wrinkle (below) makes this layering mandatory, not optional.

### B. Browser Web Speech API + Apple's enhanced Dutch voices — the correct ship-now default
- **Apple ships downloadable higher-quality Dutch voices on iOS/iPadOS.** Dutch options are **Xander (nl-NL, "ABN"/standard Dutch, male)** and **Ellen (nl-BE, Flemish, female)**; each offers **Default vs. Enhanced (a.k.a. Premium) Quality**, downloaded over Wi-Fi via *Settings → Accessibility → Spoken Content → Voices*. Enhanced "has better pronunciation and sound quality but uses more storage." [Apple Support: Adjust voice & speed, support.apple.com/en-us/111798, accessed 22 Jun 2026; AbilityNet "customise the screen-reader voice in iOS 18", mcmw.abilitynet.org.uk, accessed 22 Jun 2026]. **Action: the parent downloads the Enhanced Xander voice once on the iPad; the game selects it via `speechSynthesis.getVoices()` by matching `lang === 'nl-NL'` and the name containing "Xander".** *zeker* the voices/tiers exist.
- **Quality read:** Apple's *enhanced* Dutch voices are markedly better than the thin "default" voice the parent dislikes, and are warm enough for a children's game; they are not as natural as 2024–25 neural cloud voices, but they are free, on-device, private, and offline. *waarschijnlijk* (no neutral nl-NL MOS benchmark found; confirm by ear).
- **`speak()` requires a user gesture on iOS** — WebKit silently drops an utterance triggered from `setTimeout`/page-load; it must fire inside a tap/click handler. [TestMu/LambdaTest "Speech Synthesis API browser support", testmuai.com, accessed 22 Jun 2026]. Design the read-aloud to start from the child's tap (it already is gesture-driven), and "warm up" the synth on first tap.
- **Known iOS bugs to handle:** speech can stop when Safari is backgrounded (needs resume/restart), and the "silent switch" can mute output. [weboutloud.io "State of Speech Synthesis in Safari", accessed 22 Jun 2026 — note: that site also advertises *its own* paid "premium voices", unrelated to Apple's]. Add a visible play/replay control and a resume-on-visibility handler.
- **Cost €0; offline yes (once the voice is downloaded); fully client-side — no server, satisfies the repo's no-third-party rule.** *zeker.*

### C. The karaoke word-timing problem on iPad Safari (the crux)
- **`onboundary` (the `boundary` event) is how you'd normally drive a word highlight** — it carries a `charIndex` you map back to the word. [MDN "SpeechSynthesisUtterance: boundary event", developer.mozilla.org, accessed 22 Jun 2026 — MDN flags the feature "Limited availability… not Baseline".]
- **Reality across browsers:** Chromium fires it **per word**; **macOS Safari fires per word too** (a practitioner notes macOS Safari "will never have a `name` of `'sentence'`… you'll see an extra `'word'` event" and that **Safari does not provide `charLength` on boundary events**); **Android does not fire it at all.** [codersblock.com "JavaScript Text to Speech and its many quirks", accessed 22 Jun 2026; dev.to/jankapunkt "Cross-browser speech synthesis", accessed 22 Jun 2026.] **For iOS/iPadOS specifically, reliable word-level boundary events are *not* established** — the practitioner sources document Safari quirks and missing fields, and none demonstrate dependable per-word boundaries on iOS. **Verdict: treat iPad-Safari word boundaries as unavailable/unreliable.** *waarschijnlijk* / partly *onzeker* — worth one 5-minute on-device test, but do not architect around it.
- **Therefore the karaoke highlight on iPad must be driven by your own timing**, with two viable sources:
  1. **Estimated per-word timing (recommended default).** Compute each word's share of the utterance from a length/syllable-weighted model, scaled to the configured rate, and advance the highlight on a `requestAnimationFrame` clock started at `onstart`. This is exactly the "timed fallback" the prototype already has — **make it the primary path on iPad.** Pros: works with *any* audio engine incl. the dynamic-name lines; cons: drift on long sentences (mitigate by re-syncing per sentence/segment and keeping segments short, which also suits a dyslexic reader). *zeker* this works; tuning needed.
  2. **Pre-baked exact timing (for static lines).** When you pre-render audio with Piper/XTTS at build time, also emit **word-level timestamps** and ship them as JSON next to the audio; the highlight then reads ground-truth times. Generate timestamps with a **forced aligner**: **Echogarden** (TypeScript + WASM/ONNX, cross-platform, does TTS *and* forced alignment via DTW) is the best-fit toolchain because it can run in Node at build time and align Dutch text to the rendered audio; **aeneas** (Python/C, DTW, sub-sentence→word granularity) and **WhisperX** (wav2vec2 forced alignment, word timestamps) are alternatives. [github.com/echogarden-project/echogarden; github.com/readbeyond/aeneas; github.com/m-bain/whisperX — all accessed 22 Jun 2026.] *zeker* these produce word timings offline.
- **Bottom line for iPad:** word-perfect highlighting comes from *pre-baked timing JSON* (static lines) + *estimated timing* (dynamic/runtime lines); the browser boundary event is a nice-to-have only where it happens to work (desktop).

### D. Piper — the best free/open Dutch upgrade, with an iPad-Safari asterisk
- **License: MIT** (Rhasspy open-sourced all current models under MIT). The original **rhasspy/piper repo was archived 6 Oct 2025** (read-only) but models remain downloadable on Hugging Face and forks (e.g. `piper-plus`) continue. [github.com/rhasspy/piper VOICES.md; quick-tts.com "Every Piper Voice Ranked"; accessed 22 Jun 2026.] *zeker.*
- **Dutch voices (verified list):** **nl_NL:** `mls` (medium), `mls_5809` (low), `mls_7432` (low), `pim` (medium), `ronnie` (medium); **nl_BE:** `nathalie` (x_low, medium), `rdh` (x_low, medium). [rhasspy/piper VOICES.md, accessed 22 Jun 2026.] *zeker.*
- **Quality — important honest flag:** the most-cited Dutch voice **`nl_NL-mls-medium` is reported as poor/"gibberish" on short sentences** (trained on audiobooks, very punctuation-sensitive). Multiple users prefer the **Belgian-Dutch (nl_BE nathalie / rdh)** voices, calling them "working great." [huggingface.co/rhasspy/piper-voices discussion #13, accessed 22 Jun 2026.] For an ABN-Dutch child this is a real trade-off: Flemish accent vs. broken Netherlands-Dutch. **Audition `nl_NL-pim` and `nl_NL-ronnie` (newer, non-MLS) before assuming nl_BE is the only option.** *zeker* on the complaint; *waarschijnlijk* pim/ronnie are better than mls.
- **In-browser feasibility on iPad — the asterisk.** Piper runs as ONNX models on **WASM/CPU** (e.g. `@mintplex-labs/piper-tts-web`, which caches models in the Origin Private File System and is "frontend-only"). [github.com/Mintplex-Labs/piper-tts-web, accessed 22 Jun 2026.] **But ONNX-Runtime-Web has open, unresolved iOS Safari problems:** WASM-SIMD produces **incorrect output on iOS 16.4+** (microsoft/onnxruntime #15644, still open), and **WebGPU/JSEP mode causes severe CPU (400%+) and memory (1GB+) blow-ups crashing the tab on Safari 26.x** (microsoft/onnxruntime #26827, opened 18 Dec 2025, still open) — though the **plain non-JSEP WASM backend "works correctly in all configurations."** [both accessed 22 Jun 2026.] **So if you run Piper live in iPad Safari: force `numThreads:1`, disable SIMD/JSEP, expect first-token latency, and test on the real device.** *zeker* the bugs are open.
- **Recommended Piper integration: build-time pre-bake, not live.** Render every static line to `.opus/.mp3` with the chosen Piper voice in a Node/Python build step, align with Echogarden for word timings, and ship static assets. This dodges every iOS-Safari runtime risk and gives perfect karaoke timing — at the cost of not covering dynamic strings (handle those with the browser voice + estimated timing). *zeker.*
- **Word timing:** Piper itself does not emit word boundaries in the browser build; get timings via forced alignment at build time (Section C). *waarschijnlijk.*

### E. Kokoro — ruled out for Dutch
- **Model: Kokoro-82M, Apache-2.0**, 82M params, runs 100% client-side via Transformers.js (WebGPU/WASM). [huggingface.co/hexgrad/Kokoro-82M; npm `kokoro-js`; accessed 22 Jun 2026.]
- **No Dutch.** The base model supports 9 languages — American/British English, Spanish, French, Hindi, Italian, Japanese, Brazilian Portuguese, Mandarin — **Dutch is absent**; the browser `kokoro.js` build is **English-only** (af_/am_/bf_/bm_ voices). [github.com/hexgrad/kokoro; kokoro.js README; accessed 22 Jun 2026.] **Disqualified for nl-NL** despite its great license/size/in-browser story. *zeker.*

### F. Voice-cloning the parent's own voice (pre-baked warm lines)
The goal — lines sound like the parent without re-recording every edit — is achievable but **only as a build-time pre-bake on a GPU/CPU off-device**, then shipped as static audio. None of these can or should run live in iPad Safari, and **none can voice the runtime-inserted names** (see Section G).

- **XTTS-v2 (Coqui) — best Dutch quality, NC license (fine here).** Supports **Dutch (nl)** among 16–17 languages; **zero-shot clone from ~6 s** of reference audio (more/fine-tune improves similarity). Widely regarded as the best open local multilingual cloning engine in 2025–26. **License = Coqui Public Model License (CPML) 1.0.0: "allows only non-commercial use of a machine learning model and its outputs"; license terms must propagate to anyone you give outputs to; NO geographic/EU exclusion.** Coqui Inc. shut down Jan 2024, but the **idiap/coqui-ai-TTS fork (`pip install coqui-tts`)** keeps it running on current Python/PyTorch (code is MPL-2.0). [huggingface.co/coqui/XTTS-v2 + its LICENSE.txt; github.com/idiap/coqui-ai-TTS; accessed 22 Jun 2026.] **Verdict: fully usable for this private project; just record ~1–5 min of the parent's clean Dutch speech, pre-bake the static lines.** *zeker* license/Dutch; *waarschijnlijk* realism on a home recording.
- **OpenVoice V2 (MyShell) — cleanest license (MIT), but Dutch is cross-lingual only.** **MIT, free for commercial + research** (since Apr 2024). Natively supports EN/ES/FR/ZH/JA/KO; Dutch only via its zero-shot **cross-lingual** path (tone-color transfer onto a base speaker), so nl-NL pronunciation/naturalness is less certain than XTTS. [github.com/myshell-ai/OpenVoice; huggingface.co/myshell-ai/OpenVoiceV2; accessed 22 Jun 2026.] **Use this if you ever want a permissive (non-NC) license and can accept the Dutch quality.** *zeker* license; *onzeker* Dutch quality.
- **F5-TTS — strong tech, but model is CC-BY-NC.** **Code = MIT; pretrained model = CC-BY-NC** ("due to the training data Emilia, an in-the-wild dataset"). Zero-shot clone from ~3 s. Base model is EN/Mandarin; Dutch only via community fine-tunes (availability unverified). [github.com/SWivid/F5-TTS + LICENSE; accessed 22 Jun 2026.] NC-only like XTTS; no clear stock Dutch advantage over XTTS, so XTTS is the better default. *zeker* license; *onzeker* Dutch.
- **Piper fine-tuning to the parent's voice.** Piper voices can be **trained/fine-tuned** on a target speaker (MIT), producing a small, fast, offline ONNX voice — lighter than XTTS and pre-bakeable, but needs more clean source audio and a training run, and Dutch G2P quality inherits Piper's quirks. A good "own-voice, permissive-license, low-runtime-cost" option if you're willing to do a training pass. *waarschijnlijk.*
- **Apple Personal Voice — closest match in spirit, but unusable here.** On-device cloning from ~15 min of reading; encrypted on-device; usable by *native* apps via an authorization API. **But it is English + Mandarin (mainland) only — no Dutch — and it is an iOS/AVSpeechSynthesizer API, not exposed to Safari/Web Speech.** A web game cannot call it. [Apple Support 104993 "Create a Personal Voice"; WWDC23 "Extend Speech Synthesis with personal and custom voices"; accessed 22 Jun 2026.] **Note for the parent:** Personal Voice can be selected as a *system* Spoken-Content voice, but the game's in-page `speechSynthesis` cannot guarantee access to it — don't depend on it. *zeker.*

### G. The dynamic-text wrinkle: names inserted at runtime
The child's name ("Alvah") and a companion's name are spliced into lines at runtime, so a **pure pre-baked-audio approach cannot cover every utterance.** Options, by engine:
- **Browser voice (Web Speech):** trivially handles any string including names — synthesizes live. This is *why the browser voice stays in the mix even after a Piper/clone upgrade.* Karaoke uses estimated timing (Section C-1).
- **Pre-baked (Piper/clone):** cannot voice an unknown name. **Patterns:** (a) **hybrid** — pre-bake the fixed parts, speak the name with the live browser voice, concatenate (small timbre seam, usually acceptable for a child); (b) **pre-bake a small closed set of names** if only a handful are ever used (Alvah + the companion) — then everything is pre-baked and you splice known clips; (c) **pre-bake whole lines per known name** (cheap if the name set is tiny). **Recommended:** option (b)/(c) — the name set here is effectively two, so pre-bake each line in each needed name variant and avoid live synthesis entirely for the warm/cloned voice. *zeker* this is the clean solution given a tiny name set.

### H. Quality floor / also-rans (briefly)
- **eSpeak-NG** — open (GPL), tiny, 100+ languages incl. Dutch, runs anywhere incl. WASM; **formant synthesis = robotic**, "not as good as expected" for Dutch. Useful only as a guaranteed-available fallback, not a primary read-aloud for a child. [blog.brightcoding.dev "eSpeak NG", Sep 2025; github.com/espeak-ng; accessed 22 Jun 2026.] *zeker.*
- **MaryTTS** — open (LGPL), but **no Dutch out-of-the-box** (German/English/French/Italian/etc.). Not applicable. [search of MaryTTS supported languages, accessed 22 Jun 2026.] *zeker.*
- **Mimic 3 (Mycroft)** — Piper's lineage/sibling; superseded by Piper for this use; no advantage over Piper for Dutch. *waarschijnlijk.*

## Recommendations (staged, with thresholds)
1. **Ship-now (do this week):** Keep `speechSynthesis`. On the iPad, have the parent **download the Apple *Enhanced* Dutch "Xander" voice** (Settings → Accessibility → Spoken Content → Voices → Nederlands). In code, select it by `lang==='nl-NL'` + name match, and **make the read-aloud start from the child's tap** (gesture rule) with a visible play/replay button + resume-on-visibility handler. This alone fixes "the default voice is weak." *Threshold to move on:* if, by ear, enhanced Xander still isn't warm/clear enough for the parent.
2. **Karaoke now:** **Promote the existing "timed fallback" to the primary highlight driver on iPad** (length/syllable-weighted per-word durations on a rAF clock, re-synced per short segment). Optionally *also* listen to `onboundary` and prefer it *only if* it demonstrably fires per-word on the actual iPad. Keep segments short — good for dyslexia *and* for timing accuracy. *Threshold:* if estimation drifts noticeably on longer story beats, shorten segments or move those lines to pre-baked timing.
3. **Free/open upgrade (next):** Stand up a **build-time pre-bake pipeline**: choose the best-sounding **Piper** Dutch voice (audition `nl_NL-pim`, `nl_NL-ronnie`, and `nl_BE-nathalie`; **do not use `nl_NL-mls`**), render every *static* line to `.opus`, and run **Echogarden** forced alignment to emit per-word timing JSON. Ship audio+JSON as static assets; the `Narrator` layer plays them with ground-truth karaoke timing; dynamic-name lines stay on the browser voice. *Threshold to prefer this over Apple voice:* once a Dutch Piper voice passes the parent's ear test. *Threshold to avoid live in-browser Piper:* the open ONNX iOS bugs — only attempt live Piper on iPad with `numThreads:1`, SIMD/JSEP off, after a real-device test.
4. **Warm parent-voice (optional, highest-effort):** Record ~1–5 min of the parent reading clean Dutch; pre-bake static lines with **XTTS-v2** (NC license = fine here) — or **OpenVoice V2 (MIT)** if you want a permissive license. **Pre-bake each line in each needed name variant** (Alvah + companion) so the cloned voice never needs live synthesis. Keep the browser voice as the fallback for anything not pre-baked. *Threshold:* only if the parent wants the personal-voice warmth and is OK recording + maintaining a small build step.

## The plain-language recommendation (for a non-developer)
- **Right now:** On the iPad, go to **Instellingen → Toegankelijkheid → Gesproken materiaal → Stemmen → Nederlands** and **download "Xander" in "Verbeterde" (Enhanced) kwaliteit**. The game will use that voice — it's free, works offline, and sounds much better than the standard one. The word-by-word highlight will be driven by the game's own timer (the most reliable way on an iPad).
- **The free upgrade later:** the developer can pre-make all the fixed spoken lines with a free, open voice called **Piper** (pick the nicest-sounding Dutch one — *not* the "mls" one, which sounds broken) and store them as small audio files with exact word timings. The game swaps these in with zero change to how it looks.
- **Your own voice:** **Yes, this is possible and allowed for a private family project.** You record a few minutes of yourself reading Dutch, and a free tool (**XTTS-v2**) makes the fixed lines sound like you. It runs on a computer ahead of time (not on the iPad), and the game plays the resulting audio. Because Alvah's name and his companion's name are slotted in, the developer pre-makes each line for those specific names — so your voice covers everything that matters.
- **What won't work:** Apple's "Personal Voice" (record-your-own-voice on iPhone) sounds ideal but only speaks English/Chinese and a website game can't use it. And "Kokoro," another free voice tool, doesn't speak Dutch at all.

## The technical default for the build
`Narrator` interface with pluggable engines. **v1 engine = Web Speech (`speechSynthesis`)** pinned to enhanced **nl-NL Xander**, gesture-started, with a **self-driven word clock** as the karaoke source (boundary event used only opportunistically on desktop). **v2 engine = static pre-baked audio** (Piper now, optional XTTS parent-clone later) with **forced-aligned per-word timing JSON**. **Dynamic-name lines always route to the live browser voice (v1) with estimated timing**, or to pre-baked per-name variants when the warm voice is wanted. No servers, no third-party runtime scripts — all synthesis is either the OS voice (v1) or static files generated at build time (v2), satisfying the repo's client-side/no-third-party rules.

## Caveats
- **iPad-Safari boundary events are the biggest unverified point.** Practitioner sources show macOS Safari fires word events (minus `charLength`) and Android fires none; *no source cleanly confirms reliable per-word boundaries on iOS/iPadOS*. The safe assumption (build your own timing) is robust either way, but **a 5-minute on-device test on the actual iPad** would upgrade this from *waarschijnlijk* to *zeker*. [MDN; codersblock; dev.to — accessed 22 Jun 2026.]
- **No neutral nl-NL MOS/quality benchmark was found** comparing Apple-enhanced vs. Piper vs. XTTS — quality verdicts here are evidence-informed but ultimately **decide by ear with the parent.**
- **Apple voice availability is OS/region/version-dependent;** confirm "Enhanced Xander" actually downloads on the specific iPadOS version, and that `getVoices()` exposes it to Safari (sometimes only after a tap/`voiceschanged`). The **EU DMA delay reported for Apple *Intelligence*/Siri-AI does NOT remove Spoken-Content/VoiceOver voices** — but re-check on the device. [apple.com/newsroom "Due to DMA, Siri AI delayed in EU…", Jun 2026, accessed 22 Jun 2026.]
- **ONNX-Runtime-Web iOS bugs are open and recent** (#15644 WASM-SIMD wrong output iOS 16.4+; #26827 JSEP CPU/RAM blow-up Safari 26.x, opened Dec 2025). They are the reason to **pre-bake rather than run neural TTS live on iPad.** Re-check their status before attempting any live in-browser neural voice. [github.com/microsoft/onnxruntime, accessed 22 Jun 2026.]
- **License discipline:** XTTS-v2 (CPML) and F5-TTS model (CC-BY-NC) are **non-commercial only** — fine for this private game, but **would block any future public/commercial release**; Piper, OpenVoice V2, eSpeak-NG, Kokoro are permissive. CPML also requires propagating its terms to anyone you share outputs with — irrelevant for private family use, relevant if you ever distribute. *zeker.*
- **Cloning realism depends on the parent's recording** (clean mic, quiet room, a few minutes); a phone-in-a-noisy-room sample will sound worse. XTTS works from ~6 s but more/cleaner audio helps. *waarschijnlijk.*
- **Piper Dutch trade-off is real:** the best-documented nl voice is broken on short text, and the reliable ones are Belgian-Flemish-accented. For an ABN-Dutch child, audition `nl_NL-pim`/`ronnie` carefully before accepting a Flemish accent. *zeker* on the complaint.
- **This space moves monthly** — re-verify Piper fork health, XTTS fork maintenance, Kokoro's language list (Dutch could be added), and the ONNX/Safari bug status before launch.
